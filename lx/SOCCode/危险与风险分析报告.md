## 一、并发与线程安全

### 🔴 R-01 VehicleSignals 无锁跨线程读写

**位置**: `ADAS.py` ROS 回调 vs `_control_loop_impl`

**现象**: `VehicleSignals` 是一个普通 `@dataclass`，被 ROS 订阅回调（ROS executor 线程）写入，被 100Hz 控制循环（timer 线程）读取，**没有任何锁保护**。

```python
# ROS callback (executor thread):
def _ego_pose_callback(self, msg):
    self.signals.ego_x = x        # write
    self.signals.ego_received = True  # write

# Control loop (timer thread):
# _control_loop_impl reads self.signals.* every 10ms
```

**影响**: Python GIL 保证了单个属性赋值的原子性，但**不保证多属性一致性**。一个回调可能写了一半（比如 `ego_x` 已更新但 `ego_y` 还是旧值），控制循环就刚读到了。对于位姿 (x, y, yaw) 三个属性，可能出现半新半旧的组合，导致坐标投影错误。

**实际风险**: 单属性原子性在 CPython 实现。位姿半更新在 100Hz 下差一个周期的概率低但非零。最坏情况是投影坐标跳一拍，被低通滤波平滑后影响有限。**在安全关键系统中这是不可接受的竞态。**

**修复建议**: ROS callback 侧加锁写整个 signals 快照，控制循环侧加锁读整个快照。或在 callback 中用 `copy()` 做 double-buffer。

### 🟡 R-02 runtime 模块级全局变量的竞态

**位置**: `runtime.py` 第 24-36 行

**现象**: `configure_runtime()` 通过 `global` 语句修改模块级变量 `NANO_ROLE`, `IS_PRIMARY` 等。注释明确警告了 `from runtime import IS_PRIMARY` 的风险（会捕获旧值），但 `global` 赋值本身在多线程场景下不是原子的——如果其他模块在 configure 进行到一半时读了这些全局变量，可能看到不一致的状态。

```python
def configure_runtime(role=None):
    global _runtime_config, NANO_ROLE, IS_PRIMARY, PRIMARY_IP, SECONDARY_IP, HB_PORT, HB_GRACE_S, LOG_FILE
    _runtime_config = resolve_runtime_config(role)
    NANO_ROLE = _runtime_config.nano_role        # 赋值了一半...
    IS_PRIMARY = _runtime_config.is_primary       # 如果这里被读怎么办？
    PRIMARY_IP = _runtime_config.primary_ip
    # ...
```

**实际风险**: `configure_runtime()` 仅在 `main()` 中 `rclpy.spin()` 之前调用，此时 spin 线程还未启动。**当前安全，但如果有人未来在运行时动态切换角色则会出问题。**

### 🟡 R-03 pipeline.py 对 signals 的就地篡改

**位置**: `pipeline.py` 第 91-106 行

```python
suppress = bool(getattr(memory, 'suppress_lead_for_overtake', False))
saved_lead_received = signals.lead_received
saved_lead_last_rx = signals.lead_last_rx_time
if suppress:
    signals.lead_received = False          # ← 篡改输入！
    signals.lead_last_rx_time = -1e9      # ← 篡改输入！
# ... evaluate_lead_context ...
if suppress:
    signals.lead_received = saved_lead_received       # ← 还原
    signals.lead_last_rx_time = saved_lead_last_rx     # ← 还原
```

**现象**: "纯计算"管线函数修改了传入的 `signals` 对象，然后再改回去。这是对"纯计算"语义的违反——如果 `evaluate_lead_context` 或其下游抛异常，原始值不会被还原，导致后续周期 `lead_received` 永远为 False。

**实际风险**: 当前控制循环有 try/except 兜住异常，异常路径走 `_send_emergency_stop`，下一周期会重新从回调拿到正确的 `lead_received`。但如果异常刚好发生在 `lead_received = False` 和 `lead_received = saved_lead_received` 之间，**本周期**会错误地无前车。

**修复建议**: 不修改原始 signals，而是创建副本或传递额外参数 `suppress_lead=False`。

### 🔵 R-04 config.py globals() 注入运行时参数

**位置**: `config.py` 第 545-561 行

```python
g = globals()
for k, raw in data.items():
    if not (isinstance(k, str) and k.isupper() and k in g):
        continue
    # ...
    g[k] = new  # ← 修改模块全局变量
```

**现象**: `_load_param_overrides()` 在 import 时从 `params.yaml` 读取配置，用 `globals()` 直接修改模块全局变量。虽然做了白名单校验（只接受全大写已存在的标量名），但 `globals()` 修改对已经通过 `from config import *` 星导入的模块**不会生效**——因为星导入在 import 时就把值拷贝到了目标模块的命名空间。

**实际风险**: 如果有模块在 `config.py` 被加载后、`_load_param_overrides()` 执行前做了 `from config import LOOP_HZ`，那么覆盖不会传递到该模块。当前代码中 `_load_param_overrides()` 在模块末尾调用，import 顺序保证了它在被其他模块 import 之前执行。**但如果有人把 import 顺序搞乱就会出静默 bug。**

## 二、数值安全

### 🔴 R-05 横向控制器 dt 除法未做零保护

**位置**: `control/lateral_controller.py` 第 177、222、267 行

```python
rrate = wrap_angle(...) / dt              # 第 177 行
cte_dot = clamp(... / dt, ...)           # 第 222 行
psi_d = clamp(... / dt, ...)             # 第 267 行
```

虽然有 `dt = clamp(memory.dt, CTRL_DT_MIN, CTRL_DT_MAX)` 或 `dt = clamp(now - memory.lat_last_update_t, CTRL_DT_MIN, CTRL_DT_MAX)` 保护，`CTRL_DT_MIN` 如果配置为 0 或负数，就会导致除零。

**查证**: `CTRL_DT_MIN` 在 config.py 中定义为 `0.005`（5ms），所以正常运行不会除零。但如果有人通过 `params.yaml` 把 `CTRL_DT_MIN` 设为 0——它通过了白名单（是大写已存在标量），就会导致除零崩溃。

**修复建议**: 在除法处加 `dt = max(dt, 1e-6)` 硬兜底。

### 🟡 R-06 MPC 安全距离投影的数值边界

**位置**: `control/mpc_longitudinal.py` 第 141-145 行

```python
if dist < min_safe_dist:
    deficit = (min_safe_dist - dist) / max(min_safe_dist, 1.0)
    floor = _clamp(1.0 + 5.0 * deficit, 1.0, LON_CMD_MAX_BRAKE_DECEL)
```

**现象**: 当 `min_safe_dist < 1.0` 时（理论上不应发生，但 `compute_min_safe_distance` 的下限是 `SAFE_DIST_STANDSTILL` 约 2m），除数退化到 1.0。更关键的是 `deficit` 可以非常大——如果车距 `dist` 接近 0 而 `min_safe_dist` 约 5m，`deficit = 5/5 = 1.0`，`floor = 1 + 5 = 6 m/s²`，这是合理的。但如果 `dist` 为负值（不应发生但未防护），`deficit` 可以大于 1，`floor = 1 + 5 * (min_safe_dist + |dist|) / min_safe_dist`，制动地板值可能飙到危险水平。

**实际风险**: `dist` 来自 `x_rel`（纵向相对距离），正常情况不应为负。但上游 `LeadTracker.evaluate` 中 `x_rel = state.filtered_x_rel` 没有非负保护。如果前车坐标投影产生异常值，负 `dist` 会触发 `6+ m/s²` 的紧急制动地板值。

### 🟡 R-07 PID 控制器积分 windup 的衰减策略

**位置**: `longitudinal.py` 第 139-156 行

```python
if sat_drive or sat_brake:
    self._dist_i *= ACC_I_DECAY_SAT    # 饱和时积分衰减
else:
    self._dist_i = i_candidate          # 正常时用新积分值
```

**现象**: 饱和衰减系数 `ACC_I_DECAY_SAT` 如果配置接近 1.0（几乎不衰减），在持续饱和期间积分项不会大幅收缩，导致退出饱和后产生很大的超调。如果配置为 0（立即清零），可能导致稳态误差。

**实际风险**: 当前期默认值应该是合理的，但 params.yaml 注入机制允许运行时修改。

### 🟡 R-08 弯道速度限制中的除零

**位置**: `control/longitudinal_policy.py` 第 364 行

```python
v_curve_max = math.sqrt(CORNERING_MAX_LAT_ACCEL / max(lateral_ctx.curv_guard, 1e-6))
```

**现象**: 当 `curv_guard` 非常小时（直道），`v_curve_max` 可能变得非常大，超过物理可能速度。这不会导致除零，但可能导致巡航目标速度不合理地高。

**实际风险**: `v_tgt` 后续被 `min(DRIVER_SET_SPEED, SYSTEM_MAX_CRUISE, ROAD_LIMIT_SPEED, v_curve_max)` 限制，所以实际不会产生危险速度输出。但 `CORNERING_MAX_LAT_ACCEL / 1e-6` 如果大到 `sqrt` 超过浮点范围，理论上会产生 `inf`。

### 🟡 R-09 lead_v_proj 初始化为零的影响

**位置**: `control/context.py` ControlMemory

```python
filtered_lead_v_proj: float = 0.0
last_lead_v_proj: float = 0.0
```

**现象**: 前车投影速度的初始值为 0.0。在系统启动时、前车首次出现前，这些值都是零。如果 `filtered_lead_v_proj` 被误用（例如在前车未确认时），默认零速可能导致错误的 TTC 计算（`ego_v - 0` = 全部自车速度作为接近速度），触发误 AEB。

**实际风险**: 代码中 `lead_ctx.acc_has_lead` 门控确保了只有确认前车后才进入 ACC 模式。但 AEB 的 `apply_aeb` 函数中的 `closing` 参数计算可能有独立的前车速度路径。当前代码是安全的，但依赖多级bool门控的正确性。

## 三、控制逻辑

### 🔴 R-10 ACC 控制器 reset 时机的隐患

**位置**: `control/longitudinal_policy.py` 第 85-86 行

```python
if lead_ctx.lead_acquired:
    # ...
    managers.lon_ctrl.reset()    # ← 重置PID积分项和速度差滤波
```

**现象**: 前车重获时（`lead_acquired=True`），`lon_ctrl.reset()` 清零积分项 `_dist_i` 和速度差滤波 `_v_diff_filt`。但 `reset()` 只在**主前车重获**路径上执行，**弯道保持退出恢复到 ACC 时没有 reset**——因为 `compute_longitudinal_policy` 的弯道保持分支在退出时 `managers.lon_ctrl.reset()` 也被调用了，但**AEB告警退出恢复ACC时没有reset**。

仔细看代码弯道保持分支：
```python
elif in_curve_hold:
    # ...弯道保持逻辑...
```
退出弯道保持后自然回到 ACC 分支，此时如果前车仍然存在且 `acc_has_lead=True`，会直接进入 ACC 分支——但 `lead_ctx.lead_acquired` 是**首次确认**的边沿信号，不会在弯道保持退出后重复触发。**这意味着 PID 的积分项保留了弯道保持期间的旧值，可能导致纵向指令跳变。**

**实际风险**: 中等。弯道保持期间 PID 被清零，退出后从零开始积分比从旧值开始更安全。但旧值是 0（因为弯道保持分支也调了 `managers.lon_ctrl.reset()`），所以**当前代码实际上是安全的**——弯道保持分支确实调了 reset。但 AEB 告警保持 → 巡航 → 无前车 → 有前车重获这条路径需要仔细确认。

### 🟡 R-11 ACC 模式下的 AEB 全制动确认机制

**位置**: `control/longitudinal_policy.py` 第 279-289 行

```python
if (... and closing_speed > 0.8 and ...):
    memory.aeb_full_confirm_count = min(memory.aeb_full_confirm_count + 1, AEB_FULL_CONFIRM_CYCLES)
else:
    # 非对称退出：一次衰减 2 拍
    memory.aeb_full_confirm_count = max(memory.aeb_full_confirm_count - 2, 0)
```

**现象**: AEB 全制动需要连续多个周期确认才触发（`AEB_FULL_CONFIRM_CYCLES`），但退出时每次减 2。这意味着：在边界条件下（确认计数 = 阈值），一个周期的条件不满足就会触发衰减（减2），如果下一拍条件又满足了，又加1。这会产生**AEB全制动快速抖动**——拍1确认到N触发全制动，拍2不确认减2退出全制动，拍3又确认回来。

**实际风险**: `apply_aeb` 中 `full_confirmed` 标志检查是 `>= AEB_FULL_CONFIRM_CYCLES`，而衰减是 `-2`。假设阈值为3：
- 拍1: count=3 → full_confirmed=True → 最大制动
- 拍2: 条件不满足 → count=1 → full_confirmed=False → 减制动
- 拍3: 条件满足 → count=2 → full_confirmed=False → 制动恢复中
- 拍4: 条件满足 → count=3 → full_confirmed=True → 最大制动

这种 4 拍周期振荡在实际制动系统中是不可接受的。**但** `AEB_FULL_CONFIRM_CYCLES` 的阈值通常设得较高（≥5），使得衰减2不太容易造成反复穿越阈值。

### 🟡 R-12 超车状态机对 signals 的短路修改

**位置**: `pipeline.py` 第 94-106 行 — 已在 R-03 中讨论

但还有另一个角度：**超车期间 `signals.lead_last_rx_time = -1e9`**，这意味着超车解除后 `lead_last_rx_time` 被恢复为原始值。如果超车持续了数秒，原始时间戳可能比 `now - LEAD_TIMEOUT_S` 更早，导致前车数据在恢复瞬间被判为不新鲜（`lead_fresh=False`），触发一拍的前车丢失保护期。

**实际风险**: 设计注释明确指出这是为了让 `lead_fresh` 判定失败，防止 `raw_has_lead` 复活。恢复后的那一拍"不新鲜"会被后续周期自然修正。风险可控。

### 🟡 R-13 巡航模式超速保护逻辑

**位置**: `control/longitudinal_policy.py` 第 141 行

```python
elif ego_v >= (min(DRIVER_SET_SPEED, SYSTEM_MAX_CRUISE, ROAD_LIMIT_SPEED) - 0.2):
    lon_cmd = max(lon_cmd, 0.0)
```

**现象**: 当车速接近设定速度时，`lon_cmd` 被钳到 ≥ 0（不加速），但在不是 `acc_has_lead` 且不在巡航保护期（`cruise_drive_guard=False`）的分支中。如果此时已经快要超速但还是 `lon_cmd < 0`（加速中），强制 `max(lon_cmd, 0)` 会导致**瞬间的油门切断**，相当于阶跃输入，通过 `LonSmoothing` 后仍然可能产生突变。

**实际风险**: `LonSmoothing` 会平滑过渡。但如果平滑系数 `LON_FILTER_ALPHA` 很大（接近 1），平滑效果有限。当前默认值应该是合理的。

## 四、主备切换

### 🔴 R-14 心跳种子解析的单点故障

**位置**: `heartbeat.py` 第 28-60 行

```python
def _parse_primary_hb_fields(msg: str):
    # ...
    for token in msg.split():
        if ':' not in token:
            continue
        tag, _, val = token.partition(':')
        if tag == 'SEQ':
            try:
                seq = int(val)
            except ValueError:
                pass
            continue
        # PSI/DELTA/ACC 的解析...
    if psi is None or delta is None or acc is None:
        return None
    return psi, delta, acc, seq
```

**现象**: 如果主机心跳帧格式损坏（比如 CRC 校验码被拼进字段、编码错误等），解析可能返回 `(0.0, 0.0, 0.0, None)`——这些值是合法浮点数，会通过 `_safe_float` 校验和 range clamp，被当作有效种子写入备机控制器。

**实际风险**: 如果主机异常发出 `PSI:0 DELTA:0 ACC:0`，备机接管后会把 `lon_smooth` 重置到 0（零制动零加速）和 `last_delta` 重置到 0（方向盘归零），这在高速行驶中是最危险的——意味着一瞬间方向盘打直 + 零制动，可能在备机接管的第一拍产生致命输出。

**缓解**: `_handle_takeover_edge` 中有 range clamp（`delta_seed = clamp(delta_seed, -MAX_DELTA, MAX_DELTA)`），但 `lon_seed = 0.0` 被 clamp 到 `[-LON_CMD_MAX_DRIVE_ACCEL, LON_CMD_MAX_BRAKE_DECEL]` 范围内**是合法的**——0.0 代表零加速零制动，不触发保护。

**修复建议**: 在 `consume_takeover_seed()` 中加一个合理性检查：如果种子值全部为零（PSI=0 AND DELTA=0 AND ACC=0），视为无效种子，不执行接管初始化，让备机从自己的内部状态开始。

### 🟡 R-15 备机接管后的保护窗口期间 PID 积分项残留

**位置**: `ADAS.py` 第 246-300 行

**现象**: 接管边沿 `_handle_takeover_edge` 只重置了 `lon_smooth` 和 `memory.last_delta`，但没有重置横向 PID 的积分项 `memory.psi_i_term`。如果主机和备机的航向误差不同（它们在物理上位置略有差异），备机接管后横向积分项保留了以前自己（作为备机 standby 时）的残留值，可能导致第一拍的横向输出偏差。

**实际风险**: `TAKEOVER_GUARD_DURATION_S` 保护窗口限制了 delta 的变化率，间接限制了横向输出跳变。但积分项的残留会在保护窗口结束后仍然影响输出。

**修复建议**: 在 `_handle_takeover_edge` 中也清零 `memory.psi_i_term`。

## 五、串口与通信

### 🔴 R-16 ESP32 接收解析对标签缺失的静默容忍

**位置**: `serial_link.py` rx 线程，`common.py` `parse_tagged_lines`

```python
def parse_tagged_lines(buf, valid_tags):
    # 只解析 valid_tags 中的标签，其他行静默跳过
```

**现象**: ESP32 发送的回读帧如果某些标签缺失（比如 `P:` 缺失但 `D:` 和 `B:` 在），`esp_psi` 会保持上一拍的旧值。控制循环无法区分"ESP32 没有发送 P 字段"和"ESP32 发送了与上一次相同的 P 值"。

**实际风险**: 如果 ESP32 的航向传感器故障导致 `P:` 字段丢失，Jetson 侧会一直使用过时的航向回读值做闭环监控。**没有回读超时机制**——回读值只在 rx 线程更新，没有时间戳检查。

**修复建议**: 在 `Esp32Serial` 中增加回读值新鲜度检查（类似 `lead_last_rx_time`），超时后标记 `esp_psi_valid=False`。

### 🟡 R-17 CRC 校验不会修复帧——只有丢弃

**位置**: `control/serial_protocol.py`

**现象**: ESP32 侧校验 CRC 失败后丢弃整个帧。Jetson → ESP32 方向有 CRC 保护，但 ESP32 → Jetson 方向（回读帧）**没有 CRC**——回读帧通过 `parse_tagged_lines` 按 `P:`/`D:`/`B:` 标签解析，没有校验和。

**实际风险**: 如果 UART 线路噪声导致一行回读帧中某个标签值被篡改（如 `P:1.234` 变成 `P:1.834`），Jetson 会在闭环中使用了错误值。当前回读值仅用于监控（`pub_esp_psi` 等话题发布），不参与控制计算，所以**对控制安全无影响**。但如果未来有人用回读值做闭环反馈，这个风险就会升级。

## 六、初始化与状态管理

### 🟡 R-18 LaneWidthEstimator 冷启动过窄

**位置**: `lateral.py` 第 35-36 行

```python
self._width = LANE_WIDTH_MIN   # 3.5m
self._prev_width = LANE_WIDTH_MIN
```

**现象**: 冷启动时车道宽估计使用 `LANE_WIDTH_MIN = 3.5m`（设计意图：偏窄以触发边界修正），而默认值是 `LANE_DEFAULT_WIDTH = 3.8m`。边界余量按车道宽计算：

```python
lane_margins_from_width(width):
    half = width * 0.5
    safe = half * 0.75   # 3.5m → 1.3125m
    warn = half * 0.6     # 3.5m → 1.05m
    hard = half * 0.4     # 3.5m → 0.7m
```

在冷启动的前几秒内（`LANE_EST_MIN_SAMPLES = 60` 帧之前），边界余量偏小，意味着**边界修正会提前触发、更用力地修正**。这是保守的——早期多修正比少修正安全。但偏窄的边界也会让 ACC/AEB 的横向窗口变窄（`compute_lead_lateral_window` 依赖车道宽），可能延迟前车确认。

**实际风险**: 保守设计，风险是性能而非安全。但需要确认 `LANE_EST_MIN_SAMPLES = 60` 在 100Hz 下只需要 0.6 秒就能积累足够样本，冷启动窗口很短。

### 🟡 R-19 ControlMemory 初始全零对首拍控制的影响

**位置**: `control/context.py` ControlMemory

```python
@dataclass
class ControlMemory:
    dt: float
    cycle_count: int = 0
    psi_i_term: float = 0.0
    filtered_road_psi: float = 0.0
    # ... 全部从 0 开始
```

**现象**: 系统启动时 `filtered_road_psi = 0.0`，`filtered_cte = 0.0`，`filtered_curv = 0.0`。首拍道路航向到来后，低通滤波器从 0 跳到实际值的过程需要若干拍收敛。在此期间：
- `filtered_road_psi` 从 0 渐变到真实值 → 航向误差偏大 → 方向盘可能在首几拍偏向一侧
- `filtered_curv` 从 0 开始 → 曲率为零 → 不触发弯道相关保护

**实际风险**: `evaluate_control_health` 门控要求 `road_ready` 和 `ego_ready`，在首帧数据到来前不会进入控制解算。首帧到来后的低通收敛速度取决于 `ROAD_PSI_FILTER_ALPHA`（默认约 0.04-0.1），收敛时间约 1-3 秒。在此期间横向输出可能偏激。

**修复建议**: 在 `evaluate_control_health` 返回 `road_ready=True` 的首拍，用首帧值初始化所有低通滤波器（而非从 0 慢慢爬）。横向控制器已有类似机制：`first_update` 标志位处理首次更新。但低通滤波器本身没有。

## 七、逻辑与边界

### 🟡 R-20 超车状态机的 IDLE → WAIT 条件可能误触发

**位置**: `control/overtake.py`

```python
# WAIT 进入条件
if (
    abs(ego_v) < OVT_EGO_STILL_V   # 自车静止
    and abs(lead_v_proj) < OVT_LEAD_STILL_V  # 前车静止
    and 0 < lead_x_rel < OVT_TRIGGER_DIST_M  # 前方接近
    and lead_x_rel > OVT_TRIGGER_MIN_DIST_M   # 但不太近
    ...
):
```

**现象**: 如果 `OVT_EGO_STILL_V` 和 `OVT_LEAD_STILL_V` 阈值设置不当（太大），在低速跟车场景下（例如堵车蠕行，车速 0.3 m/s）可能误判为"静止前车"而触发超车。

**实际风险**: 取决于参数值。`OVT_LEAD_STILL_V` 和 `OVT_EGO_STILL_V` 在 config.py 中定义，需要仔细审核。

### 🟡 R-21 AEB 告警状态机的 cooldown 与 AEB 制动逻辑的交互

**位置**: `control/aeb_alert.py` 第 66-72 行

```python
if self.active and (now - self.start_t) > AEB_ALERT_HOLD_TIME_S:
    # 告警超时退出
    self.active = False
    self.has_lead = False
    self.hold_speed = 0.0
    self.stop_hold_until = 0.0
    self.cooldown_until = now + 2.0
```

**现象**: AEB 告警退出后进入 2 秒冷却期。在冷却期内 `self.armed = False`（第 48 行没执行因为 `now < cooldown_until`），所以**冷却期内即使检测到新的前车丢失也不会再次进入告警**。如果前车在这 2 秒内反复丢失，车辆可能在无保护的情况下低速行驶。

**实际风险**: 低概率。AEB 告警本身是低速保持（不加速），冷却期内如果前车重新出现，告警状态机会重新 `armed=True`。如果前车反复丢失，真正的 AEB（基于 TTC/距离的紧急制动）仍然有效——AEB 告警和 AEB 制动是两套独立逻辑。

## 八、配置与构建安全

### 🟡 R-22 params.yaml 运行时覆盖的安全隐患

**位置**: `config.py` 第 515-570 行

**现象**: `_load_param_overrides()` 在 import 时执行，允许通过 `params.yaml` 覆盖任何全大写已存在的标量常量——包括 `MAX_DELTA`, `LON_CMD_MAX_BRAKE_DECEL`, `LOOP_HZ` 等安全关键参数。

**没有范围限制**：安全限幅类参数（如 `MAX_DELTA`）可以通过 YAML 被改大到物理驱动器无法承受的值，导致方向盘转角超限。虽然代码注释了"只允许标量"，但没有区分"调参类参数"和"安全限幅类参数"。

**修复建议**: 建立安全参数白名单，只允许调节控制增益（K_PSI_P, ACC_KD 等），禁止修改安全限幅（MAX_DELTA, LON_CMD_MAX_BRAKE_DECEL 等）。或者分两层：`safe_overrides` 和 `all_overrides`，前者只包含增益类参数。

### 🔵 R-23 LOOP_HZ 变更不会更新已有 dt 派生值

**位置**: `config.py` 第 134-146 行

```python
def _resolve_loop_hz() -> int:
    try:
        hz = int(float(os.environ.get('LOOP_HZ', '100')))
    except (TypeError, ValueError):
        hz = 100
    return max(10, min(200, hz))

LOOP_HZ = _resolve_loop_hz()
```

**现象**: `LOOP_HZ` 可以通过环境变量在 10-200 之间设置。但代码中有一些模块硬编码了与频率相关的参数（如 `LANE_EST_MIN_SAMPLES = 60` 注释写的是"样本数"而非秒数），如果频率从 100Hz 改到 50Hz，同样的 60 个样本代表 1.2 秒而非 0.6 秒，时间窗口倍增。

**实际风险**: `LaneWidthEstimator` 的 `_window_size` 方法已经按频率调整了窗口：`secs * loop_hz`。大多数时间相关参数在 config.py 中以秒为单位定义。但 `LEAD_CONFIRM_CYCLES` 和 `AEB_FULL_CONFIRM_CYCLES` 等纯周期数参数如果依赖隐含的 100Hz 假设，在 50Hz 下确认时间翻倍。

### 🔵 R-24 Python 3.6 兼容性——dataclasses backport 依赖

**位置**: `requirements.txt`

```
dataclasses>=0.8; python_version < "3.7"
```

**现象**: 代码大量使用 `@dataclass` 和 `@dataclass(frozen=True)`。在 Python 3.6 上，这些依赖 `dataclasses` backport。如果 Jetson 上没有安装这个包（`pip install` 失败或被跳过），所有 `from dataclasses import ...` 都会触发 `ImportError`，**整个节点无法启动**。

**实际风险**: 低。requirements.txt 明确声明了依赖，正常部署流程会安装。但如果有人手动启动而跳过了 `pip install -r requirements.txt`，就会遇到这个问题，且错误信息可能不明确。
